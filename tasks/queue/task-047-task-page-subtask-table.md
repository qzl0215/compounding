# 任务 task-047-task-page-subtask-table

## 任务摘要

- 短编号：`t-047`
- 父计划：`memory/project/operating-blueprint.md`
- 任务摘要：
  把任务页收成单一子任务表格清单
- 为什么现在：
  当前任务页把执行清单拆成多段卡片、阶段说明和重复文案，用户只需要一个清楚的子任务表格，不需要离散内容。
- 承接边界：
  只收任务页展示层：去掉多段阶段卡片、diff 摘要区块和已发布折叠区，只保留一个按阶段排序的子任务表格；不改阶段模型，不改首页和发布页。
- 完成定义：
  用户打开 `/tasks` 后，第一屏只看到一个子任务表格清单；表格按阶段排序，已发布事项不再在这页平铺，细节只在行展开后查看。

## 执行合同

### 要做

- 把任务页从多区块结构改成单一表格页。
- 补一个轻量排序读模型，把当前子任务按待验收、执行中、待执行、待规划排序。
- 把阶段信息收进表格列，而不是继续用整段说明卡表达。
- 保留行展开详情，但把页面级离散内容全部拿掉。

### 不做

- 不改首页。
- 不改发布页。
- 不新增新的任务状态或阶段规则。
- 不把已发布历史继续留在任务页首屏。

### 约束

- 继续复用现有 `DemandStage` 与 `TaskDeliveryRow`，不新增新的长期状态源。
- 任务页只做执行清单，不重新长成说明书或工作台。
- 主表优先展示子任务本身，机器事实只能放在展开态。

### 关键风险

- 如果只删区块不补排序，用户会失去阶段感知。
- 如果把阶段信息塞回大段文案，页面还是会继续显得离散和啰嗦。

### 测试策略

- 为什么测：这轮是任务页结构收口，容易把阶段顺序、显示字段或展开态带坏。
- 测什么：子任务排序逻辑、released 过滤、任务页表头和展开信息、现有 lint/test/build/preview 探活。
- 不测什么：不为纯文案细节补快照测试，也不改发布页或首页测试。
- 当前最小集理由：只锁表格视图的核心判断和排序规则，避免为视觉小改动堆重复测试。
