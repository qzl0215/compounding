# 任务 task-046-diff-based-test-roi-selection

## 任务摘要

- 短编号：`t-046`
- 父计划：`memory/project/operating-blueprint.md`
- 任务摘要：
  用改动范围驱动 test 与验证选择，降低门禁膨胀
- 为什么现在：
  当前仓库已经有静态、构建、运行时和 AI 输出门禁，但测试与验证仍主要靠固定顺序和人工经验挑选；继续扩门禁数量的 ROI 已经下降。
- 承接边界：
  只在现有验证体系上增加“按 diff 选检查”的显式逻辑与退休建议，不新建 eval 平台、不引入 LLM judge、不增加新的长期状态源。
- 完成定义：
  系统能根据改动范围给出推荐测试集合、选择原因和退休建议；静态 / 构建 / smoke / AI 输出门禁继续保留，但不再靠堆数量表达质量。

## 执行合同

### 要做

- 定义 touchfiles / dependency / module ownership 到测试集合的映射规则。
- 在现有验证链上增加 `SelectedChecks` 与选择原因的轻量读模型。
- 补“过重测试的退休建议”，避免旧测试长期堆积。

### 不做

- 不建设 LLM judge 平台。
- 不增加新的验证层。
- 不为了追求覆盖率而给每个改动都加测试。

### 约束

- 默认优先级保持：静态门禁 → 构建/集成 → 运行时 smoke → 用户可感知体验验收 → AI 输出门禁。
- 只能基于现有对象与脚本扩展，不新增新的状态源。
- 选测逻辑必须短、可解释、可回退。

### 关键风险

- 如果 touchfiles 规则太粗，测试推荐会失真。
- 如果退休建议没有明确边界，可能误删仍有保护价值的测试。

### 测试策略

- 为什么测：这条主线本身就是测试 ROI 优化，必须验证选测结果是否稳定且不会漏掉关键回归。
- 测什么：diff 到 check selection 的映射、fallback 逻辑、退休建议条件、既有门禁兼容性。
- 不测什么：不为每个模块生成巨量矩阵测试，也不引入新的评估平台。
- 当前最小集理由：只锁最常见的改动类别和最关键的回退逻辑，先保证高频高价值场景稳定。
