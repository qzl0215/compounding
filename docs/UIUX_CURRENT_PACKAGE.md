# 当前 UIUX 资料包

## 文档用途

这份文档是给资深 UIUX AI 专家的单文档资料包。

目标不是介绍“这个仓库里有哪些文件”，而是让专家快速理解：

- 当前产品到底是什么
- 当前 UI 在服务什么认知链路
- 首页、任务页、文档页、发布页分别承担什么职责
- 哪些体验已经有价值，哪些问题最值得重构
- 哪些系统边界不能被设计改动破坏

阅读完这份文档，专家应能直接进入 UIUX 重构，而不需要先在仓库里自行考古。

## 产品定位

### 产品一句话

这是一个面向 AI 长期协作的操作系统式项目门户。

它不是一个传统 SaaS 管理后台，也不是一个普通文档站，而是一个把规则、路线图、运营蓝图、任务、记忆、代码索引和发布状态收拢到统一认知入口的 AI-Native Repo Portal。

### 当前产品角色

当前产品承担 4 种角色：

| 角色 | 说明 |
| --- | --- |
| 经营驾驶舱 | 首页用一页高浓度摘要呈现当前项目的方向、战役、组织和风险 |
| 项目管理清单 | 任务页把 task 与 Git 状态联动起来，成为轻量项目管理面板 |
| 文档工作台 | 文档页既是阅读器，也是 Markdown 直编与 AI 重构入口 |
| 发布控制台 | 发布页展示当前 release、运行态、回滚入口与发布历史 |

### 当前产品不是什么

- 不是通用企业后台
- 不是重型工单系统
- 不是多人协同写作平台
- 不是低代码页面搭建器
- 不是数据库驱动的 CMS

当前系统刻意保持“文件即真相源，UI 负责读取、呈现、轻量编辑和受控操作”。

## 当前目标用户

### 主用户

1. 仓库 owner / 项目负责人  
需要快速看懂项目当前方向、主线、风险和发布状态。

2. 参与项目的 AI agent  
需要快速建立上下文，知道先看什么、再做什么、在哪里回写。

3. 资深设计/产品/架构协作者  
需要快速理解当前系统的认知结构，而不是从零读整仓库。

### 非目标用户

- 完全不了解仓库、也不参与项目治理的一般浏览者
- 需要复杂权限体系和审批流的企业协同团队

## 当前 UX 要解决的核心问题

当前 UI 的核心使命不是“让用户点很多功能”，而是解决 5 个更重要的问题：

1. 减少理解成本  
让人和 AI 都能快速知道这个项目是什么、当前重点是什么、下一步读哪里。

2. 减少上下文切换  
把路线图、战术蓝图、任务、文档、发布状态放到同一个门户里。

3. 减少真相源漂移  
页面信息尽量直接读取 Markdown 和 release registry，而不是再复制出一套后台数据。

4. 让改动有边界  
task、Git、文档、记忆、发布状态形成闭环，而不是“改了就改了”。

5. 让问题可解释  
页面异常、版本漂移、服务没启动、样式失效，都应能被明确解释，而不是靠人猜。

## 信息架构总览

当前前台公开的主入口只有 4 个：

| 页面 | 路径 | 当前职责 |
| --- | --- | --- |
| 首页 | `/` | 经营驾驶舱与认知总览 |
| 任务 | `/tasks` | 任务清单与 Git 联动状态 |
| 文档 | `/knowledge-base` | Markdown 阅读、编辑、AI 重构 |
| 发布 | `/releases` | release 历史、本地运行态、回滚入口 |

### 当前主导航特征

- 左侧为固定侧栏
- 顶部没有二级全局导航
- 首页没有右侧目录
- 任务页、文档页、发布页保留右侧粘性页内导航

### 当前首页承担的双重角色

首页不是“营销首页”，而是：

- 项目当前状态的一页压缩镜像
- 进入深层页面和深层文档的导航总入口

因此首页的质量直接决定整个系统是否容易理解。

## 页面级说明

## 首页：经营驾驶舱

### 当前目标

首页要在一页内回答 5 个问题：

1. 这个项目为什么存在
2. 现在最重要的目标是什么
3. 当前里程碑和战术拆解是什么
4. 谁负责什么
5. 当前有哪些认知资产和风险需要关注

### 当前首页模块

当前首页固定为 5 个模块：

1. 使命 / 愿景 / 价值主张
2. 战略路线与下一里程碑
3. 当前战役与运营蓝图
4. 组织职责矩阵
5. 关键认知资产与风险

### 当前首页优点

- 模块数量已被压缩，没有继续做成仪表盘噪声墙
- 结构顺序接近一个合理的项目叙事链
- 关键入口按钮已集中在首页
- 首页数据来自 Markdown 真相源，不是手工拼的假数据

### 当前首页问题

- 文字仍然偏重，专家视角下“信息密度高但扫描效率不够高”
- 卡片数量仍偏多，模块内视觉节奏还可继续收敛
- 首页虽然有叙事链，但“经营驾驶舱感”还不够强，尤其缺少更强的优先级层次
- “当前战役与运营蓝图”是信息价值最高的模块，但视觉权重还没有拉开
- “关键认知资产与风险”里同时承载了系统入口和风险信息，信息语义仍有轻微混杂

### 当前首页对专家最重要的设计命题

- 如何让首页更像“经营驾驶舱”，而不是“高质量卡片文档页”
- 如何在不引入更多模块的前提下，进一步提升扫描效率
- 如何让“当前最重要的事”拥有更强的视觉支点
- 如何让首页既是导航，又不显得像文件树入口

## 任务页：轻量项目管理清单

### 当前目标

任务页负责把 task 从纯 Markdown 文件提升为可读、可追踪、可与 Git 状态联动的轻量 PM 面板。

### 当前展示方式

当前任务页以表格为主：

| 列 | 说明 |
| --- | --- |
| Task | 任务标题与路径 |
| 目标 | 任务目标 |
| 状态 | todo / doing / blocked / done |
| Git | 分支、最近提交、合并状态、漂移状态 |
| 关联模块 | 当前任务涉及的模块 |
| 更新痕迹 | 对 memory / index / roadmap / docs 的回写说明 |
| 打开 | 直接打开任务 Markdown |

### 当前优点

- 已摆脱卡片堆，进入更适合项目管理的表格视图
- task 状态与 Git 状态已经联动
- 能直接发现状态漂移，例如“任务 done 但未合并”或“已合并但任务未完成”

### 当前问题

- 表格信息完整，但缺少更强的“任务排序逻辑”和“视觉优先级提示”
- 对非工程用户来说，Git 状态字段仍稍显技术化
- “更新痕迹”是系统闭环关键点，但当前只在行内压缩展示，重要性表达还不够

### 对专家的关键问题

- 如何让表格继续保持专业，但阅读门槛更低
- 如何在不引入重型 PM 系统的前提下，让任务清单更像“经营面板”而不是纯工程列表

## 文档页：阅读器 + 编辑器 + AI 重构入口

### 当前目标

文档页是当前系统里最复杂、也最有潜力的页面。它同时承担：

- Markdown 阅读
- 文档目录导航
- 语义入口导航
- 正文富文本直编
- 高级模式全文编辑
- 两步式 AI 文档重构
- prompt 文档管理

### 当前交互层次

文档页当前有 3 个模式：

| 模式 | 作用 |
| --- | --- |
| 阅读模式 | 富文本阅读 |
| 编辑模式 | 在当前视觉界面中直接编辑正文 |
| 高级模式 | 修改完整 Markdown、frontmatter、managed block |

### 当前 AI 重构流程

当前 AI 重构已不是简单的一键改写，而是两步法：

1. AI 先提 3 到 7 个关键补充问题
2. 用户补充后，AI 再按强度档位重构正文结构和语言

### 当前优点

- 已经摆脱“左边源码、右边预览”的初级模式
- 默认编辑正文层，避免把 frontmatter 和托管标记暴露给日常使用
- AI 重构流程已经可用，而且有 prompt 真相源
- 同一页面同时支持阅读、编辑、AI 重构和高级模式，工作流闭环较完整

### 当前问题

- 仍然更像“工程型阅读器”，还不够像真正成熟的内容编辑工作台
- 富文本直编虽可用，但细腻度和愉悦感仍不足
- AI 重构面板的体验逻辑还偏工具感，不够“编辑助理感”
- prompt 编辑和正文编辑都在文档页内，信息层级对新用户仍偏重

### 对专家的关键问题

- 如何让文档编辑更像“专业内容工作台”，而不是“高级开发工具”
- 如何让 AI 重构过程更自然、更可信、更低打断
- 如何在不做重量 WYSIWYG 编辑器的前提下，把直编体验再提升一个量级

## 发布页：本地生产运行态与 release 管理

### 当前目标

发布页不是传统 CI/CD 控制台，而是当前仓库在本地语境下的 release registry 和运行态说明页面。

它负责回答：

- 当前激活版本是什么
- 本地生产是否真的在跑
- 运行的是哪个 release
- 是否发生了版本漂移
- 可以回滚到哪个版本

### 当前优点

- 已经能区分“页面坏了”和“服务没启动”
- 能显示 release 历史、当前运行态、端口状态、当前激活版本
- 已具备最小回滚入口

### 当前问题

- 对纯产品/设计视角来说，发布页仍有明显工程气质
- “运行态 / 版本态 / 回滚动作”的重要性层次可以更清晰
- 虽然当前已经可解释，但状态表达还可以更一眼化

### 对专家的关键问题

- 如何把发布页做得更像一个“可信赖的运行状态中心”
- 如何在保留工程精度的同时，提升非工程用户的理解效率

## 全局视觉语言

### 当前视觉方向

当前视觉是深色、高对比、带轻微科幻感的“控制台 / 指挥舱”风格。

核心特征：

- 深色底
- 带径向渐变和网格背景
- 大圆角卡片
- 半透明面板
- 强调高亮色作为导航和状态色
- 统一的黑蓝色调与微量暖色辅助

### 当前有效的视觉资产

- 统一的深色基底和光晕背景
- 卡片边框和半透明层次
- 状态标签与重点按钮
- Markdown 阅读器中的自然层级排版

### 当前视觉短板

- 某些页面的信息层级还主要靠卡片堆叠，而不是更明确的版面主次
- 首页“经营驾驶舱”虽然概念正确，但视觉上仍偏内容门户
- 页面之间风格统一度不错，但角色差异化还不够明显
- 编辑模式与阅读模式的切换，在感知上还不够“专业内容工作台”

## 当前内容与真相源架构

当前 UI 的一个关键优点是：大多数展示内容都直接来自 Markdown 和 release runtime，而不是额外造一层数据库或中间后台。

### 当前关键真相源

| 真相源 | 当前职责 |
| --- | --- |
| `AGENTS.md` | 高频执行主源 |
| `memory/project/current-state.md` | 当前状态 |
| `memory/project/roadmap.md` | 战略路线与里程碑 |
| `memory/project/operating-blueprint.md` | 当前战术蓝图 |
| `tasks/queue/*.md` | 任务清单 |
| `docs/ORG_MODEL.md` | 组织职责 |
| `code_index/*` | 模块与依赖导航 |
| release registry + local runtime state | 版本与运行态 |

### 这对 UIUX 重构意味着什么

设计方案必须尊重以下约束：

- UI 不应要求引入数据库作为前提
- UI 不应把 Markdown 真相源复制成另一套展示数据
- UI 可以做更强的提取、聚合和呈现，但不能制造新的真相漂移

## 当前最值得保留的 UX 价值

如果要重构，以下东西不应被轻易丢掉：

1. Markdown 仍是唯一内容真相源  
这是系统长期可维护性的核心。

2. 首页是一页式高浓度总览  
方向正确，只是当前还可以更专业。

3. 任务与 Git 联动  
它是 task 闭环的关键抓手。

4. 文档页支持直编与 AI 重构  
这是系统未来最有复利的能力之一。

5. 发布页可解释“为什么页面出问题”  
这是从工程系统走向可靠系统的关键一步。

## 当前最值得专家优先解决的问题

如果只能优先改 5 件事，我建议专家聚焦：

1. 首页的信息密度与主次关系  
让“当前最重要的事”更一眼化。

2. 文档页的编辑工作台质感  
让直编和 AI 重构更自然、更像专业内容工具。

3. 任务页的可读性  
在不变成重型 PM 系统的前提下，更像项目经营清单。

4. 发布页的运行态表达  
让运行中、漂移、未启动、异常更一眼可辨。

5. 全局视觉层级  
减少“卡片堆”感，让系统更像一套成熟产品，而不是多个优秀局部页面的集合。

## 不能破坏的系统约束

以下约束应视为 UIUX 重构的硬边界：

- 不能把系统做成大公司式重型后台
- 不能新增数据库或新的真相源
- 不能把 task 变成审批流
- 不能把文档系统改成完全依赖重量编辑器
- 不能破坏 `main + release/current + rollback` 的发布模型
- 不能破坏 Markdown、task、memory、index、release registry 之间的现有闭环

## 对资深 UIUX AI 专家的直接任务说明

如果把这份资料交给专家，我希望专家优先输出的不是高保真视觉稿，而是：

1. 对当前信息架构的重构判断  
哪些模块保留，哪些模块合并，哪些模块应降权。

2. 首页经营驾驶舱重构方案  
如何让首页真正成为高浓度、专业、可导航的一页仪表盘。

3. 文档工作台重构方案  
如何让“阅读 / 编辑 / 高级模式 / AI 重构”更自然地统一。

4. 任务页重构方案  
如何让 task + Git 更专业、更易读。

5. 视觉系统方向  
如何在保留深色指挥舱风格的前提下，进一步提升成熟度和可读性。

## 一句话总结

当前 UI 已经不是“零散页面”，而是一个可工作的 AI 项目操作系统门户；但它还处在“结构正确、能力可用、体验有潜力，但尚未被资深 UIUX 方法论充分打磨”的阶段。

资深专家最应该做的，不是推翻当前结构，而是基于现有真相源和现有闭环，把它打磨成一个真正专业、可信、易读、可持续演进的 AI Operating System 界面。
