# 工作模式

## 定位

- 角色决定长期职责归属，回答“谁负责什么”
- 工作模式决定当前工作入口，回答“现在处于哪种业务状态”
- 同一个角色可以参与多个工作模式
- 同一个工作模式可能涉及多个角色协同
- 详细模式定义以本文为准；角色定义以 `docs/ORG_MODEL.md` 为准

## 业务链

`需求提出 → 战略澄清 → 方案评审 → 工程执行 → 质量验收 → 发布复盘`

## 高频工作链入口

### 规划链

- 入口：路线图、运营蓝图、成功标准或发布标准不清晰，或存在目标边界争议。
- 输入：新需求或新问题、当前 `roadmap`、当前 `operating-blueprint`、尚未解开的关键问题。
- 输出：清晰的里程碑定义、更新后的 `memory/project/roadmap.md`、更新后的 `memory/project/operating-blueprint.md`、共商规划 task。
- 退出条件：已形成可执行的里程碑、蓝图和规划 task。

### 执行链

- 入口：task 已确认，方案边界已定，准备进入代码或文档落地。
- 输入：已确认的 task、相关模块上下文、必要的 code_index 与记忆。
- 输出：代码改动、文档 / 记忆 / 索引回写、可审查提交。
- 退出条件：结果达到可验收状态，且相关回写已完成。

### 交付链

- 入口：实现已完成，验收标准已知，准备验证、发布或回滚。
- 输入：已实现结果、验收标准、设计要求、运行时与测试结果、release 准备。
- 输出：通过 / 不通过结论、release 切换或回滚结果、经验沉淀与复盘结论。
- 退出条件：质量验收结论明确，版本切换或回滚完成，复盘已落地。

### 切换原则

- 弱 agent 先按这三条链路切模式，不靠经验猜。
- 规划链只在方向、边界或标准不清时进入。
- 执行链只在 task 已明确时进入。
- 交付链只在可验收或可发布时进入。

## 战略澄清

- 模式目标：
  - 在路线图、运营蓝图、成功标准或发布标准不清晰时，先把方向、边界与取舍对齐。
- 输入：
  - 新需求或新问题
  - 当前 `roadmap`
  - 当前 `operating-blueprint`
  - 尚未解开的关键问题
- 输出：
  - 清晰的里程碑定义
  - 更新后的 `memory/project/roadmap.md`
  - 更新后的 `memory/project/operating-blueprint.md`
  - 共商规划 task
- 基本原则：
  - 先把问题说清楚，再决定是否进入实现
  - 只收口关键方向，不把 roadmap 写成流水账
- 限制条件：
  - 不直接进入实现
  - 不越过用户自行补齐关键业务判断
- 进入条件：
  - 里程碑不清晰
  - 蓝图无法拆成关键子目标
  - 发布标准或成功定义模糊
- 退出条件：
  - 已形成可执行的里程碑、蓝图和规划 task

## 方案评审

- 模式目标：
  - 在动手前把方案边界、范围、验收标准和重构取舍收口到可执行水平。
- 输入：
  - 已澄清的目标和边界
  - 候选方案
  - 验收需求
  - 相关模块上下文
- 输出：
  - 方案结论
  - 范围与范围外
  - 验收标准
  - task 目标、约束与计划
- 基本原则：
  - 先定边界，再定实现
  - 只保留最关键的约束，不把评审做成审批流
- 限制条件：
  - 不直接替代架构或工程写实现细节
  - 不把评审文档膨胀成第二套 roadmap
- 进入条件：
  - 需求已澄清，准备进入实现
  - 存在多个方案或边界争议
- 退出条件：
  - task 已可执行
  - 验收标准与约束已明确

## 工程执行

- 模式目标：
  - 在最小必要上下文里实现、删减和重构，把确认过的方案安全落地。
- 输入：
  - 已确认的 task
  - 方案边界
  - 相关代码、模块、索引与记忆
- 输出：
  - 代码改动
  - 文档 / 记忆 / 索引回写
  - 可审查提交
- 基本原则：
  - 只构建最小上下文
  - 新增逻辑必须伴随清理
- 限制条件：
  - 不在没有 task 边界时直接开工
  - 不只加不删
- 进入条件：
  - task 已明确
  - 方案边界和验收标准已确定
- 退出条件：
  - 结果达到可验收状态
  - 相关回写已完成

## 质量验收

- 模式目标：
  - 用事实判断结果是否满足方案、设计和运行要求，并明确剩余风险。
- 输入：
  - 已实现结果
  - 验收标准
  - 设计要求
  - 运行时与测试结果
- 输出：
  - 通过 / 不通过结论
  - 风险说明
  - task 状态更新建议
- 基本原则：
  - 以事实为准，不凭感觉判断完成
  - 只做关键验证，不把检查扩成形式主义
- 限制条件：
  - 不为了量化而量化
  - 不跳过关键 smoke、测试或运行态检查
- 进入条件：
  - 工程执行已完成
  - 结果准备交付或合并
- 退出条件：
  - 形成明确验收结论
  - 通过后才能进入发布复盘

## 发布复盘

- 模式目标：
  - 把已通过验收的结果并入 `main`、准备 release、执行切换或回滚，并沉淀关键经验。
- 输入：
  - 已通过验收的结果
  - 运行态状态
  - release 准备结果
  - 风险说明
- 输出：
  - `main` 合并结果
  - release 切换或回滚结果
  - 经验沉淀与复盘结论
- 基本原则：
  - 发布与回滚必须串行
  - 经验沉淀只记录关键复用结论
- 限制条件：
  - 不跳过验证直接切换版本
  - 不用 `git reset` 伪装回滚
- 进入条件：
  - 质量验收已通过
  - release 已准备完成
- 退出条件：
  - 版本切换或回滚完成
  - 复盘与经验沉淀已落地
