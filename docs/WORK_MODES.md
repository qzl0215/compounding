# 工作模式

## 定位

- 需求环节优先于 task 与工作模式入口；若事情仍处于 `待思考 / 待规划`，不得直接切到工程执行
- 工作模式只描述输入、输出、进入退出条件，不承载静态身份标签

## 业务链

`需求提出 → 战略澄清 → 方案评审 → 工程执行 → 质量验收 → 发布复盘`

### 规划链

- 入口：方向成立但边界、取舍、优先级、成功标准或验收标准未清
- 输入：新需求或新问题、当前 `roadmap`、当前 `operating-blueprint`、尚未解开的关键问题
- 输出：更新后的单层 plan、清晰的体验验收标准、必要时产出的 task
- 退出条件：已完成“扩选项 → 收决策 → 产出 task”

### 执行链

- 入口：task 已确认，方案边界已定，准备进入代码或文档落地
- 输入：已确认的 task、相关模块上下文、必要的 code_index 与记忆
- 输出：代码改动、文档 / 记忆 / 索引回写、可审查提交
- 退出条件：结果达到可验收状态，且相关回写已完成

### 交付链

- 入口：实现已完成，验收标准已知，准备验证、发布或回滚
- 输入：已实现结果、验收标准、设计要求、运行时与测试结果、release 准备
- 输出：通过 / 不通过结论、release 切换或回滚结果、经验沉淀与复盘结论
- 退出条件：质量验收结论明确，版本切换或回滚完成，复盘已落地

### 切换原则

- 先判断需求处在哪个环节，再决定是否切到某个工作模式
- 规划链只在方向、边界或标准不清时进入
- 执行链只在 task 已明确时进入
- 交付链只在可验收或可发布时进入

## 战略澄清

- 入口：路线图、运营蓝图、成功标准或发布标准不清晰
- 输入：新需求或新问题、当前 `roadmap`、当前 `operating-blueprint`、尚未解开的关键问题
- 输出：清晰的方向与边界、更新后的 `memory/project/roadmap.md` 与 `memory/project/operating-blueprint.md`、是否进入 task 的明确判断
- 退出条件：已形成可执行的 plan 和下一批 task 边界

## 方案评审

- 入口：需求已澄清，准备进入实现，或存在多个方案 / 边界争议
- 输入：已澄清的目标和边界、候选方案、验收需求、相关模块上下文
- 输出：方案结论、范围与范围外、验收标准、task 执行合同
- 退出条件：task 已可执行，验收标准与约束已明确

## 工程执行

- 入口：task 已明确，方案边界和验收标准已确定
- 输入：已确认的 task、方案边界、相关代码 / 模块 / 索引 / 记忆
- 输出：代码改动、文档 / 记忆 / 索引回写、可审查提交
- 退出条件：结果达到可验收状态，且相关回写已完成

## 质量验收

- 入口：工程执行已完成，结果准备交付或合并
- 输入：已实现结果、验收标准、设计要求、运行时与测试结果
- 输出：通过 / 不通过结论、风险说明、task 状态更新建议
- 退出条件：形成明确验收结论；通过后才能进入发布复盘

## 发布复盘

- 入口：质量验收已通过，release 已准备完成
- 输入：已通过验收的结果、运行态状态、release 准备结果、风险说明
- 输出：`main` 合并结果、release 切换或回滚结果、经验沉淀与复盘结论
- 退出条件：版本切换或回滚完成，复盘与经验沉淀已落地
