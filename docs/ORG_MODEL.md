# 组织模型

## 设计原则

- 组织角色是稳定职责镜头，不是官僚部门
- 新增角色只有在相邻角色连续超载或职责冲突时才允许
- 角色长期低频或职责重叠时，优先合并，不保留空壳组织
- 组织设计职责并入 `总经办 / Foreman Office`，不单列 `HR`
- 规则服务于效率，不服务于自我扩张；若规则本身拖慢复利效率，可直接简化

## 工作模式

### 战略澄清模式

- 使命：在路线图、运营蓝图或关键发布标准不清晰时，先把方向和边界对齐，再进入执行。
- 适用场景：
  - 里程碑定义不清
  - 运营蓝图无法拆出关键子目标
  - 用户目标、价值判断或成功标准仍模糊
- 主要产物：
  - 更新后的 `memory/project/roadmap.md`
  - 更新后的 `memory/project/operating-blueprint.md`
  - 共商规划 task
- 不该做什么：
  - 不在规划未清时直接进入实现
  - 不把 roadmap 写成流水账

### 方案评审模式

- 使命：在动手前收口问题定义、方案边界、验收标准与重构取舍，避免“做了很多但做偏了”。
- 适用场景：
  - 新需求刚进入
  - 结构性改动准备开始
  - 设计、产品、架构边界需要统一
- 主要产物：
  - 清晰的 task 目标、范围和验收标准
  - 方案结论与边界说明
  - 必要的规则或文档更新建议
- 不该做什么：
  - 不越过架构边界直接规定实现细节
  - 不把评审过程做成审批流

### 工程执行模式

- 使命：在最小必要上下文里实现、删减和重构，把方案安全落地。
- 适用场景：
  - task 已明确，开始编码或重构
  - 需要做局部实现、模块收敛、文档回写
  - 需要在短分支上完成可验证改动
- 主要产物：
  - 代码改动
  - task / docs / memory / code_index 回写
  - 可审查的提交
- 不该做什么：
  - 不只加不删
  - 不在没有 task 边界时直接开工

### 质量验收模式

- 使命：用事实验证结果是否满足方案、设计和运行要求，并明确剩余风险。
- 适用场景：
  - 实现完成后准备交付
  - 需要做页面、运行时或规则回归
  - 结果是否应标记为完成存在争议
- 主要产物：
  - 测试与检查结果
  - 风险说明
  - task 状态更新建议
- 不该做什么：
  - 不为了量化而量化
  - 不把检查扩成形式主义流程

### 发布复盘模式

- 使命：把已经通过 review 的结果切入 `main`、准备 release、执行切换或回滚，并沉淀关键经验。
- 适用场景：
  - 准备把已完成任务并入 `main`
  - 需要执行 release 切换或回滚
  - 发布后要复盘为什么成功或失败
- 主要产物：
  - release 记录
  - 回滚或切换结果
  - 经验沉淀与复盘结论
- 不该做什么：
  - 不跳过验证直接切换版本
  - 不用 `git reset` 伪装回滚

## 决策层

### 总经办 / Foreman Office

- 使命：保持主线清晰、节奏稳定，确保系统始终像创业团队而不是大公司。
- 主要职责：
  - 决定当前主线、优先级和发布裁决
  - 负责组织设计、角色增删合并和规则收敛
  - 判断哪些规范该保留，哪些规范该删减
- 主要产物：
  - `AGENTS.md`
  - `memory/project/roadmap.md`
  - `memory/project/operating-blueprint.md`
  - 关键 ADR 与发布决策
- 何时介入：
  - 方向切换、优先级冲突、角色重叠、发布风险或规则失效时
- 不该做什么：
  - 不把任务管理做成审批流
  - 不把组织设计扩成多层部门树

## 交付层

### PMO

- 使命：把每次改动变成边界清晰、可追踪、可回写的 task。
- 主要职责：
  - 维护 task 清单与状态
  - 追踪更新痕迹和交付闭环
  - 组织复盘，确保经验被沉淀而不是丢失
- 主要产物：
  - `tasks/queue/*.md`
  - `tasks/archive/*`
  - `memory/project/operating-blueprint.md`
  - 复盘相关更新痕迹
- 何时介入：
  - 任何可合并改动开始前、交付中、交付完成后
- 不该做什么：
  - 不发明 lane、泳道、工时审批等重制度
  - 不替代产品、架构或工程做专业判断

### 产品

- 使命：把问题定义清楚，把价值判断和验收边界说清楚。
- 主要职责：
  - 定义需求、目标、范围与验收标准
  - 判断改动是否真正提升主线效率
  - 保证方案不是“做很多”，而是“做对重点”
- 主要产物：
  - task 的目标、范围、验收标准
  - `memory/project/operating-blueprint.md` 中的子目标和发布标准
  - 方案摘要与价值判断
- 何时介入：
  - 新需求进入、优先级竞争、验收标准不清时
- 不该做什么：
  - 不越过架构边界直接指定实现细节
  - 不把 roadmap 写成流水账

### 设计

- 使命：让信息架构、交互和视觉表达服务于快速理解。
- 主要职责：
  - 设计首页与关键页面的信息编排
  - 提供更优 UI/UX 方案与验收视觉标准
  - 避免“看起来专业但实际更难用”的设计
- 主要产物：
  - 页面信息架构
  - UI/UX 方案
  - 关键页面的视觉和交互要求
- 何时介入：
  - 首页、门户、阅读体验、结构变化或交互障碍出现时
- 不该做什么：
  - 不脱离内容真相源单独维护一套展示数据
  - 不为了炫技引入重型设计系统

## 研发层

### 架构

- 使命：让系统持续保持小模块、清边界、低耦合、易于 AI 理解。
- 主要职责：
  - 设计模块边界、接口契约、依赖方向
  - 收敛重复结构和巨型文件
  - 规划重构路线，控制复杂度扩张
- 主要产物：
  - `docs/ARCHITECTURE.md`
  - `code_index/*`
  - `module.md`
- 何时介入：
  - 结构重构、模块拆分、跨层依赖或大文件治理时
- 不该做什么：
  - 不把抽象层做成自我目的
  - 不把稳定规则和一次性经验混写

### 工程

- 使命：在最小上下文里把方案高质量落地，并持续删减旧逻辑。
- 主要职责：
  - 按 task、方案和验收标准实现改动
  - 删除被替代逻辑、收敛兼容层
  - 准备 release、执行切换或回滚修复
- 主要产物：
  - 代码改动
  - 发布准备与切换结果
  - 对应 task / docs / index 更新
- 何时介入：
  - 方案确认后进入实现、修复、重构和发布阶段时
- 不该做什么：
  - 不在没有 task 边界时直接开工
  - 不只加不删，留下长期兼容债

## 保障层

### 质量与度量

- 使命：用事实验证结果是否正确，用量化判断投入是否值得。
- 主要职责：
  - 验收结果与设计/方案是否匹配
  - 跟踪回归、风险、技术债和发布健康度
  - 量化评估工作量、成果和 ROI
- 主要产物：
  - 测试结果
  - 发布状态与风险清单
  - 技术债与量化结论
- 何时介入：
  - 验收、回归、发布前后、争议较大或 ROI 不明时
- 不该做什么：
  - 不把检查扩成形式主义流程
  - 不为了量化而量化，忽略主线价值

## 文化底线

- 规则服务于效率，不服务于扩张
- task 是边界，不是审批流
- roadmap 只记录主线变化
- 记忆只记录可复用经验和明确裁决
- 角色只是职责分工，不是组织层级炫耀
- 若规范本身限制复利效率，可直接简化规范
